# Indexing

## KV-Stores vs Relational Databases

Relational databases support 3 main disk operations:
1. **Full Scan** — read entire dataset (no index)
2. **Point Query** — lookup a specific key using an index
3. **Range Query** — retrieve a range of keys

- Range queries generalize point queries.
- Relational databases can be built on top of **KV-stores**.

---

## Hash Tables

### Pros
- Fast point lookups

### Cons
- No support for sorting → **cannot do range queries**
- Resizing is expensive (naive: **O(n)**)

→ Not suitable for most database use cases.

---

## B-Trees (Balanced N-ary Trees)

### Key Idea
- A **balanced multi-way tree** (not just binary)
- Operations: **O(log n)** for search and updates
- Supports **range queries efficiently**

### Why B-Trees over Binary Trees?

1. **Less Space Overhead**
   - Binary trees: ~1–2 pointers per node
   - B-trees: many keys share one parent → fewer pointers overall

2. **Faster in Practice**
   - Better cache usage due to modern CPU architecture

3. **Less Disk I/O**
   - Trees are shorter (fewer levels)
   - Nodes fit better into disk pages → fewer reads

---

## LSM-Trees (Log-Structured Merge Trees)

### Structure
- Data is stored in **multiple levels**
- Each level:
  - Sorted
  - Split into multiple files
- Lower levels = larger data (exponentially growing)

---

### Queries

**Point Query**
1. Start at top level
2. If not found → search next level

**Range Query**
- Merge results from all levels
- Higher levels override lower levels

---

### Updates

1. Insert into **top level (small, fast)**
2. If full → merge into next level
3. Levels grow exponentially in size

---

### Analysis

#### Queries
- Each level is sorted → binary search
- Range queries → sequential reads (efficient)

#### Updates
- Writes are cheap (small top level)
- Merging is **sequential I/O** (efficient)
- Frequent small merges at top levels
- Large merges at lower levels (expensive)

- Merging replaces overlapping data in lower levels
- Done in background, but can cause **high I/O load**

---

## When to Use What

- **B-Trees**
  - Used in most **relational databases**
  - Good for balanced read/write workloads

- **LSM-Trees**
  - Used in **NoSQL databases**
  - Optimized for **write-heavy systems**

---

## Key Takeaways

- Indexing enables fast queries (point + range)
- Hash tables are fast but too limited
- B-Trees optimize for:
  - reads
  - disk efficiency
- LSM-Trees optimize for:
  - writes
  - sequential I/O

- Choice depends on workload:
  - Read-heavy → B-Tree
  - Write-heavy → LSM-Tree
