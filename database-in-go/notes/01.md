# Files vs Databases

## 1. Saving Data to Files (Basic Approach)

### Example Idea
Open a file → overwrite everything → write new data.

### Problems

1. **Deletes old data immediately**
   - The file is wiped (truncated) before new data is written.
   - If something goes wrong mid-write → data is lost.

2. **Not safe for concurrent access**
   - If another process reads while writing → it may see half-written data.

3. **Not truly saved to disk yet**
   - When you write, data often goes to OS memory (cache) first.
   - If the system crashes → data might disappear.

---

## 2. Atomic Renaming (Safer Approach)

### Idea
Instead of writing directly:
1. Write to a temporary file  
2. Rename it to the real file  

### Why This Helps
- Renaming is **atomic** (all-or-nothing).

- If a crash happens before rename:
  - The original file is still safe.

### Remaining Problems
- Data might still be in memory (not yet on disk).
- File metadata (like size) might be saved before actual content.

### Result
- Files can still get corrupted after a crash (e.g., files filled with zeros).

---

## 3. fsync (Forcing Data to Disk)

### Idea
Call `fsync` after writing.

### What It Does
- Forces the OS to actually write data to disk.
- Reduces risk of losing data during a crash.

### Problems
- What about directory metadata?
- You may also need to `fsync` the directory.

### Why This Matters
- The process becomes complex.
- Easy to implement incorrectly.

---

## 4. Append-Only Logs

### Idea
Instead of overwriting, always add new data at the end.

### Advantages
- No overwriting → safer.
- Less risk of corruption.
- Simpler than rename-based methods.

### Problems
- **Slow searching**
  - You must scan everything (no structure).

- **No easy deletion**
  - Data keeps growing forever.

---

## 5. Why Databases Exist

All the problems above explain why databases are used.

### Databases Solve

- **Atomic writes**
  - Data is written safely (all-or-nothing).

- **Crash safety**
  - Handles syncing and recovery automatically.

- **Efficient queries**
  - Uses indexes (fast lookup instead of scanning everything).

- **Data management**
  - Handles updates, deletes, and storage cleanup.

---

## Key Takeaways

- Writing directly to files is simple but unsafe.
- Fixing it manually (rename + fsync) gets complicated fast.
- Append-only logs are safer but limited.

### Databases Exist Because They:
- Handle all these edge cases correctly.
