# B-Tree: The Ideas (Simplified Notes)

## 1. Intuition (From BST → B-Tree)

- Start from **balanced binary search trees (BSTs)**
  - BSTs store sorted data
  - Must stay balanced for good performance

- Problem:
  - Binary trees are not efficient for disk (too many small nodes)

- Solution:
  - Use **B-Trees (n-ary trees)** → each node stores **multiple keys + children**
  - Better use of **disk pages (I/O units)**

---

### How B-Trees Stay Balanced

Unlike BSTs (rotation-based balancing):

- **If node is too big** → split into 2 nodes
- **If node is too small** → merge with sibling
- All **leaf nodes stay at same height**

---

## 2. Intuition via Nested Arrays

Start with:
- A **sorted array**
  - Fast search: O(log n)
  - Slow insert: O(n)

### Improve it:
- Split into smaller arrays

### How it works:
1. Find correct sub-array → O(log n)
2. Search inside sub-array → O(log n)

- Insert becomes faster (~O(√n))

### Insight:
- Add more levels → becomes like a **tree**
- This is basically how **B-Trees work**

---

## 3. B-Tree Operations (Using B+ Tree)

### Note:
- **B+ Tree** = values only stored in **leaf nodes**
- Internal nodes = keys + pointers

---

### Insertion

1. Insert key into a **leaf node (sorted list)**
2. If node is too big:
   - Split into 2 nodes
   - Each gets half the keys
3. Update parent:
   - Replace old pointer with new ones
4. If parent becomes too big:
   - Split again (propagates upward)

### Tree Growth
- If root splits → create a **new root**
- Tree becomes taller

---

### Deletion

- Reverse of insertion

1. Remove key
2. If node too small:
   - Merge with sibling
3. If root has only 1 child:
   - Replace root with that child

### Tree Shrinks
- Height can decrease

---

## 4. Structure of Internal Nodes

Each internal node contains:
1. **Pointers to children**
2. **Keys**
   - Each key = first key of its child

---

## 5. Immutable B-Trees (Important Concept)

### Idea
- Never modify data in place
- Instead:
  - Create new copies when updating

### Example (Insert):
1. Create new leaf node with updated data
2. Parent must also be updated → copied
3. Repeat until root

→ Entire path is copied  
→ Old version still exists

---

### Why This Is Useful

1. **Safer (No Corruption)**
   - Old data is untouched
   - Crashes don’t destroy existing data

2. **Better Concurrency**
   - Readers use old version
   - Writers create new version
   - No conflicts

---

## Key Takeaways

- B-Trees are **optimized for disk storage**
- Nodes store **multiple keys** → fewer levels → less I/O
- Balanced via **split + merge**, not rotations
- B+ Trees store data only in **leaves**
- Immutable approach:
  - safer
  - better for concurrent systems

- Big idea:
  - B-Trees = scalable version of BST for real-world storage
